Computing the Diameter in any dimension: Part I

The diameter of an nnn point set X⊂RdX \subset \mathbb{R}^dX⊂Rd is a fundamental geometric primitive. It is defined as the maximum Euclidean distance between any two points:

D(X)=max⁡x,x′∈X∥x−x′∥2D(X) = \max_{x, x' \in X} \lVert x - x' \rVert_2 D(X)=x,x′∈Xmax​∥x−x′∥2​

More generally, one may start with an arbitrary metric space (X,dX)(X, d_X)(X,dX​) and use the same definition with the metric dXd_XdX​. In this three part series, I'll introduce a way to compute D(X)D(X)D(X) efficiently for non-pathological inputs—in ≈O(nh)\approx O(nh)≈O(nh) time and O(n)O(n)O(n) memory, where h<<nh << nh<<n for non-pathological inputs—independent of the dimension ddd of XXX. The method is extremely fast, uses very little working memory, does not involve any complicated data structures, and is extendeable to any metric.

Knowing the diameter of XXX conveys a few things about the point set. Some examples:

  1. The smallest enclosing sphere has radius R(X)R(X)R(X) satisfying R(X)≤12D(X)d2(d+1)R(X) \leq \frac{1}{2}D(X) \sqrt{\frac{d}{2(d+1)}}R(X)≤21​D(X)2(d+1)d​​ (Jung's Theorem)
  2. 1nD(X)≤ϵ∗(X)\frac{1}{n} D(X) \leq \epsilon_\ast(X)n1​D(X)≤ϵ∗​(X) where ϵ∗(X)\epsilon_\ast(X)ϵ∗​(X) is twice the minimum radius needed to ensure ϵ\epsilonϵ-neighbor graph Nϵ(X)N_\epsilon(X)Nϵ​(X) is connected.
  3. Let A,B⊆XA, B \subseteq XA,B⊆X. Then D(A∪B)≤D(A)+D(B)+dH(A,B)D(A \cup B) \leq D(A) + D(B) + d_H(A, B)D(A∪B)≤D(A)+D(B)+dH​(A,B), where dH(A,B)d_H(A, B)dH​(A,B) is the Hausdorff distance
  4. Let (X,dX)(X, d_X)(X,dX​) and (Y,dY)(Y, d_Y)(Y,dY​) denote two metric spaces. Then

    12∣D(X)−D(Y)∣≤dGH(X,Y)≤12max⁡{D(X),D(Y)}\frac{1}{2} \lvert D(X) - D(Y)\rvert \leq d_{GH}(X, Y) \leq \frac{1}{2}\max\{D(X), D(Y)\} 21​∣D(X)−D(Y)∣≤dGH​(X,Y)≤21​max{D(X),D(Y)}

    where dGHd_{GH}dGH​ is the Gromov-Hausdorff distance ( introduction video if you're curious )

As an aside, ϵ∗(X)\epsilon_\ast(X)ϵ∗​(X) can also be interpreted as the largest edge in the metric minimum spanning tree, which is useful itself in a variety of circumstances as this is the minimum radius one needs to ensure Nϵ(X)N_\epsilon(X)Nϵ​(X)—the union of balls centered at points in XXX with radii ϵ\epsilonϵ—is connected.

By itself, the diameter is generally not a very informative quantity. Nonetheless, it is a useful quantity to have, as its often used as a building block other algorithms. Despite this, it's not immediately clear how to compute D(X)D(X)D(X) efficiently. Let's look at some ways to go about doing just that.

Just take the max?

In Python, if XXX is a n×dn \times dn×d numpy matrix and ddd is thought of as a constant, we have a trivial one-liner that takes O(n2)O(n^2)O(n2) time and O(n2)O(n^2)O(n2) memory (if you can import SciPy's pdist):

diameter = max(pdist(X))

A more memory efficient option using O(1)O(1)O(1) memory and O(n2)O(n^2)O(n2) time is to as before enumerate all distances, but this time keep a running max:

n, diam = X.shape[0], 0.0
for i in range(n):
  for j in range(i+1, n):
    diam = max(diam, np.linalg.norm(X[i,:] - X[j,:]))

Both options require same number of distance computions; the latter is much more memory efficient. Despite this, the O(n2)O(n^2)O(n2) memory one-liner solution from SciPy is almost surely going to be much faster.

Indeed, a quick benchmark with ∣X∣=100\lvert X \rvert = 100∣X∣=100 shows the SciPy solution takes 2.59 seconds to execute 100 times while the SciPy solution takes just 0.035 seconds—a 74x improvement. This is largely due to the high latency associated with memory-access in Python and the vectorization benefits an optimized, low-level implementation is bound to have. Thus, for very small point sets, the naive yet vectorized solution is likely the way to go (if you can afford the memory!).

Obviously, O(n2)O(n^2)O(n2) can be quite expensive in both time and memory for larger nnn. Representing distances using 64-bit floats, 10k10k10k points already incurs an overhead of ≈50M\approx 50\mathrm{M}≈50M distance computations and ≈381\approx 381≈381 MB of working memory. This is simply unacceptable.

Learning from lower dimensions

Ideally we want an O(nlog⁡n)O(n \log n)O(nlogn) solution that uses O(n)O(n)O(n) memory. There is actually a simple solution that uses the following observation:

Any two points x,x′∈Xx, x' \in Xx,x′∈X satisfying dX(x,x′)=D(X)d_X(x, x') = D(X)dX​(x,x′)=D(X) must lie on the boundary of the convex hull of XXX

This is a useful observation, because it means that if we can compute the convex hull C(X)\mathcal{C}(X)C(X) of XXX in O(nlog⁡n)O(n \log n)O(nlogn) time, then perhaps we can introduce some procedure which determines D(X)D(X)D(X) using only the points on the boundary of C(X)\mathcal{C}(X)C(X). This is usually a much smaller set.

Indeed, this was one of they key observations that inspired the rotating calipers method, which can be used to compute the diameter of a convex polygon in O(n)O(n)O(n) time. The idea is here is to generate antipodal points by (figuratively) rotating calipers around the vertices of the convex hull until all antipodal points are found. D(X)D(X)D(X) is then found by taking the largest of these.

Here's two pictures to illustrate the intuition behind the rotating calipers method, borrowed from this great blog post. I recommend checking out said post if you're curious how it works. The left picture shows a (non-convex) polygon, and the right-side shows the minimum bounding box around different sets of vertices on the convex hull. Notice that D(X)D(X)D(X) can be derived from these as well, and that these boxes only intersect points on the convex hull itself.

Since the convex hull can be computed in O(nlog⁡n)O(n \log n)O(nlogn) time, e.g. using SciPys port of QHull, the rotating calipers method yields an efficient method for d=2d = 2d=2.

Despite the ingenuity of the rotating calipers method, it's not immediately clear how to translate it to higher dimensions. There exists 3D generalizations, however they seem to not be used in practice. Nonetheless, one may still compute the Convex Hull of XXX and then resort to the brute force method: if C(X)\mathcal{C}(X)C(X) has hhh points on its boundary, this method takes O(nlog⁡n+h2)O(n \log n + h^2)O(nlogn+h2) time, which can be very efficient if h<<nh << nh<<n.

Here's some example code:

from scipy.spatial import ConvexHull # Note this is just a port of QHull
diameter = max(pdist(X[ConvexHull(X).vertices,:]))

How well does this fare? Let's do a quick test with 5k uniformly random samples.

import timeit 
import numpy as np
X = np.random.uniform(size=(5000,2))
diameter_hull = lambda: max(pdist(X[ConvexHull(X).vertices,:]))
diameter_brute = lambda: max(pdist(X))

timeit.timeit(diameter_hull, number=30)
# >> 0.0359 (seconds)

timeit.timeit(diameter_brute, number=30)
# >> 19.023 (seconds)

diameter_hull() == diameter_brute()
# >> True

The naive brute-force computation, which was 74x faster than the [very naive] raw python implemention, is 533x slower than the convex hull solution. This shouldn't be surprising though: indeed, a order of magnitude difference in complexity should yield a (roughly) order of magnitude difference in runtime. Since QHull is among the most highly optimized softwares on the planet, we get the best of both worlds: the convex hull shrinks the problem size significantly, and composing with brute-force solution on the vertices of the hull leverages some vectorization benefit, which is much simpler and likely faster than a generic Python implementation of the rotating calipers method if the number of such vertices is small.

So we're done! The diameter_hull code shown above is very efficient (for d=2d = 2d=2...)!

The secret nobody tells you

It is often the case in computational geometry that one only considers cases where d=2d = 2d=2, as many geometric truths are verifiable by hand when working with the plane. Many geometric algorithms which are proven to work in O(nlog⁡n)O(n \log n)O(nlogn) only do so in the d=2d = 2d=2 or very low ddd setting: the complexity statements often hide a constant-factor which grows exponentially (or worse!) as d→∞d \to \inftyd→∞.

The convex hull is one such computation. The size of the convex hull C(X)\mathcal{C}(X)C(X) of an nnn-point set XXX scales as follows in ddd:

∣C(X)∣∼Ω(n⌊d/2⌋)\lvert \mathcal{C}(X) \rvert \sim \Omega(n^{\lfloor d/2 \rfloor}) ∣C(X)∣∼Ω(n⌊d/2⌋)

Oof. Let's see how this actually affects the computation—perhaps it won't be so bad. Let's generate a small set of 1.5k normally distributed random points in ddd dimensions for increasing ddd and profile the time it takes to compute D(X)D(X)D(X) 30 times.

for d in [2, 3, 4, 5, 6, 7, 8, 9, 15]:
  X = np.random.normal(size=(1500, d))
  time_hull = timeit.timeit(diameter_hull, number=30)
  time_naive = timeit.timeit(diameter_naive, number=30)

Oof. When the 15 dimension test case didn't complete in 30 minutes, I called it. While the brute force stays effectively constant in ddd, the convex hull approach quickly becomes intractable. Thus, even though the asymptotics in low-dimensions are great, the vectorized brute-force code (which works in any dimension) is a much more consistent choice in practice for arbitrary dimensions.

It is a fun exercise, though non-trivial, to try to imagine how to extend techniques like the rotating calipers to higher dimensions. Intuitively, if we want to do less distance computations, one option is to try to only consider points that lie on the peripheary. Points near the center of XXX will likely lie in the interior of C(X)\mathcal{C}(X)C(X), so there is no reason to consider them in the computation of D(X)D(X)D(X).

A series of clever observations

In 2001, Malandain and Boissonatt (1) published a way of getting at D(X)D(X)D(X) quickly using a few clever observations.

Let p,q∈Xp, q \in Xp,q∈X denote a pair of points satisfying D(X)=dX(p,q)D(X) = d_X(p,q)D(X)=dX​(p,q), i.e. a pair whose distance yields the diameter of XXX. Note there may be several of these, though we just need one; call these maximal pairs.

  1. If x,x′∈Xx, x' \in Xx,x′∈X are any two points such that dX(x,x′)<D(X)d_X(x, x') < D(X)dX​(x,x′)<D(X), then at least one of (p,q)(p,q)(p,q) lies outside of the ball B[x,x′]B[x, x']B[x,x′], where:

B[x,x′]=B(x+x′2,dX(x,x′)2)B[x, x'] = B\left( \frac{x + x'}{2}, \frac{d_X(x, x')}{2} \right ) B[x,x′]=B(2x+x′​,2dX​(x,x′)​)

  1. If x,x′∈Xx, x' \in Xx,x′∈X are any two points such that X/ B[x,x′]=∅X /\ B[x,x'] = \emptysetX/ B[x,x′]=∅, then D(X)=dX(x,x′)D(X) = d_X(x, x')D(X)=dX​(x,x′)
  2. Let F(x)⊂XF(x) \subset XF(x)⊂X denote the set of points of XXX furthest away from x∈Xx \in Xx∈X, and let:

    H={(x,x′):x∈F(x′),x′∈F(x)}H = \{(x, x') : x \in F(x'), x' \in F(x) \} H={(x,x′):x∈F(x′),x′∈F(x)}

    denote the set of so-called double normals. Since these pairs are each in the others furthest point sets, they must lie on the boundary of the convex hull. Thus H⊂C(X)H \subset \mathcal{C}(X)H⊂C(X).
Let's see what each of these observations afford us.

Observation #1

If x,x′∈Xx, x' \in Xx,x′∈X are any two points such that dX(x,x′)<D(X)d_X(x, x') < D(X)dX​(x,x′)<D(X), then at least one of (p,q)(p,q)(p,q) lies outside of the ball B[x,x′]B[x, x']B[x,x′], where:

B[x,x′]=B(x+x′2,dX(x,x′)2)B[x, x'] = B\left( \frac{x + x'}{2}, \frac{d_X(x, x')}{2} \right ) B[x,x′]=B(2x+x′​,2dX​(x,x′)​)

The idea underlying this simple observation is that if we find a ball B[x,x′]B[x,x']B[x,x′] covering a large portion of XXX for some pair x,x′∈Xx, x' \in Xx,x′∈X and X/ B[x,x′] = ∅X /\ B[x,x'] \, \cancel= \, \emptysetX/ B[x,x′]=​∅, then we know the set X/ B[x,x′]X /\ B[x,x']X/ B[x,x′] contains at least one of the points making up a maximal pair. Thus, if we only want to find one point in a maximal pair, we may eliminate all points in X∩B[x,x′]X \cap B[x,x']X∩B[x,x′] from consideration. For example, consider the pair of points whose bounding sphere is the red sphere shown above. Clearly the red-ball does not cover XXX, thus we know at least one point of a maximal pair exists outside of this ball. This fact itself can lead to removing large chunks of unnecessary computations done in the brute force case.

Observation #2

If x,x′∈Xx, x' \in Xx,x′∈X are any two points such that X/ B[x,x′]=∅X /\ B[x,x'] = \emptysetX/ B[x,x′]=∅, then D(X)=dX(x,x′)D(X) = d_X(x, x')D(X)=dX​(x,x′)

The second idea gives a partial stopping condition for the diameter computation. If we find a pair of points whose corresponding ball B[x,x′]B[x, x']B[x,x′] contains all of XXX, then we are done: these two points must have distance dX(x,x′)=D(X)d_X(x, x') = D(X)dX​(x,x′)=D(X). However, the converse is not necessarily true, as demonstrated by the second picture. In general, it may be case that X/ B[p,q]X /\ B[p,q]X/ B[p,q] is non-empty.

Thus, we need more than this condition alone to find a maximal pair.

Observation #3

Let F(x)⊂XF(x) \subset XF(x)⊂X denote the set of points of XXX furthest away from x∈Xx \in Xx∈X, and let HHH denote the set of double normals, defined as:

H={(x,x′):x∈F(x′),x′∈F(x)}H = \{(x, x') : x \in F(x'), x' \in F(x) \} H={(x,x′):x∈F(x′),x′∈F(x)}

Since these pairs are mutually in the others furthest point sets, they must lie on the boundary of the convex hull. Thus H⊂C(X)H \subset \mathcal{C}(X)H⊂C(X).

This observation is essentially a more refined version of the convex hull observation and provides an intuitive generalization of the the pairs of antipodal points the rotating calipers method is based on to higher dimensions. Observe that a pair (p,q)(p, q)(p,q) with distance dX(p,q)=D(X)d_X(p, q) = D(X)dX​(p,q)=D(X) matching the diameter of XXX must be a double normal: if there was any point further away from ppp than qqq is, then that pair would have a distance larger than D(X)D(X)D(X) (and vice versa). Thus, both ppp and qqq must belong to HHH.

Both the convex hull C(X)\mathcal{C}(X)C(X) and the set HHH of double-normals are shown in the right-most image above. The black lines traverse the boundary of C(X)\mathcal{C}(X)C(X), the green line connects the maximal pair whose distance yields the diameter, and the orange lines demarcate other double-normals. Observe H⊂C(X)H \subset \mathcal{C}(X)H⊂C(X), thus if we can find double-normals quickly, we may be able to find the diameter quickly as well.

Conclusions: Part I

Amazingly, the short list of ideas and observations described so far are all that is needed to produce an efficient algorithm for computing the diameter of a point set in ddd dimensions in ≈O(nh)\approx O(nh)≈O(nh) time. The algorithm is simple, but does require more of an explanation than given so far—I defer its full description until Part II, which I will cover in my next posting.
